## 1. Спецификации на входы

**К входам относятся:**

* Ввод данных пользователем через интерфейс приложения (формы, команды, кнопки).
* Структурированные данные в формате JSON, поступающие от сервера.
* Файлы конфигурации приложения.
* Авторизационные данные (логин/UserName-Telegram).
* Действия пользователя: запуск приложения, создание задачи, назначение исполнителя.
* Сетевая среда — наличие интернет-соединения.

**Характеристики входов:**

* Формат данных: JSON, текстовые строки UTF-8.
* Диапазоны значений: длина текстовых полей до 512 символов.
* Протоколы: HTTPS для обмена с сервером.
* Требуемые предварительные состояния: наличие зарегистрированного пользователя.

## 2. Спецификации на выходы

Выходы определяют результаты, которые тестируемая система должна сгенерировать:

* Ответы API в формате JSON.
* Изменённые состояния приложения (созданная задача, обновлённая история).
* Сообщения интерфейса: уведомления, ошибки ввода, подтверждения.
* Логи действий пользователя.
* Метрики выполнения (время реакции, корректность результата).

**Каждый тест должен явно приводить:**

* полученный объект данных,
* отображаемый текст,
* ожидаемый статус операции.

## 3. Среда тестирования

* **Операционная система:** Windows 10, 11 / macOS Sequoia  
* **Оборудование:** ПК с 8+ ГБ RAM, процессор не ниже Intel i5 / M1.  
* **ПО:**
    * Node.js v18+ (при тестировании фронтенда),
    * Docker (опционально),
    * браузер Chrome 125+,
    * система управления версиями Git.
* **Инструменты тестирования:**
    * Jest / Go testing — модульные тесты,
    * Postman — тестирование API,
    * Cypress — e2e-тесты,
    * системные логи.

## 4. Специальные требования

* Тестирование проводится на чистой базе данных (reset).
* Должна быть обеспечена стабильная сеть не ниже 5 Мбит/с.
* Все тесты выполняются под тестовым пользователем.
* Время реакции сервера — не более 1 секунды для стандартных запросов.
* Недопустимо ручное вмешательство в систему во время выполнения тестов.

## 5. Взаимозависимости

* Тест 1 должен выполняться первым, так как формирует первоначальное состояние.
* Тест 2 зависит от корректного выполнения Теста 1.
* Свойство A внутри каждого теста должно быть проверено до проверки Свойства B.
* Все тесты должны выполняться на одной и той же версии сборки.

## 6. Тесты

### 6.1. Тест 1 — Создание команды

#### 6.1.1. Свойство A — Корректность создания

**Ожидаемое поведение**  
Система должна создать новую команду, сохранить её в базе данных и вернуть подтверждение.

**Ожидаемая производительность**  
Время ответа — не более 500 мс.

**Ожидаемая достоверность**  
Название команды и параметры совпадают с переданными данными.

**Замечания**  
Тест выполняется на пустом списке команд.

#### 6.1.2. Свойство B — Обработка некорректных данных

**Ожидаемое поведение**  
Если имя команды пустое или содержит недопустимые символы — вернуть ошибку, команду не создавать.

**Ожидаемая производительность**  
Ответ — не более 300 мс.

**Ожидаемая достоверность**  
Никакая команда не добавляется в базу.

### 6.2. Тест 2 — Добавление участников в команду

#### 6.2.1. Свойство A — Корректность добавления

**Ожидаемое поведение**  
Участник успешно добавляется в команду, система возвращает актуальный состав команды.

**Ожидаемая производительность**  
Ответ — не более 400 мс.

**Ожидаемая достоверность**  
Список участников содержит нового пользователя.

#### 6.2.2. Свойство B — Добавление уже существующего участника

**Ожидаемое поведение**  
Система должна вернуть ошибку "участник уже в команде".

**Ожидаемая производительность**  
Ответ — до 300 мс.

**Ожидаемая достоверность**  
Состав команды не меняется.

### 6.3. Тест 3 — Создание заявки

#### 6.3.1. Свойство A — Корректность создания заявки

**Ожидаемое поведение**  
Система создаёт заявку с указанным описанием, задачей и автором. Заявка становится доступна в общем списке.

**Ожидаемая производительность**  
Ответ — не более 500 мс.

**Ожидаемая достоверность**  
Все поля заявки совпадают с входными параметрами.

#### 6.3.2. Свойство B — Ошибочные данные при создании заявки

**Ожидаемое поведение**  
При пустом описании или отсутствии задачи система должна вернуть ошибку.

**Ожидаемая производительность**  
Ответ — до 300 мс.

**Ожидаемая достоверность**  
Никакая заявка не создаётся.

### 6.4. Тест 4 — Автоматическое назначение заявки после дедлайна

#### 6.4.1. Свойство A — Корректность автопереназначения

**Ожидаемое поведение**  
Если заявка не взята до наступления дедлайна, система автоматически назначает её случайному участнику.

**Ожидаемая производительность**  
Назначение выполняется в течение 1 с после наступления дедлайна.

**Ожидаемая достоверность**  
В заявке появляется назначенный исполнитель.

#### 6.4.2. Свойство B — Ошибка при отсутствии участников

**Ожидаемое поведение**  
Если в команде нет участников, система должна зафиксировать ошибку и оставить заявку без исполнителя.

**Ожидаемая производительность**  
Проверка — до 300 мс.

**Ожидаемая достоверность**  
Статус заявки остаётся неизменным.

### 6.5. Тест 5 — Голосование за подтверждение отчёта

#### 6.5.1. Свойство A — Корректность голосования

**Ожидаемое поведение**  
Все участники могут оценить отчёт. При единогласном подтверждении заявка считается выполненной.

**Ожидаемая производительность**  
Обработка голосов — не более 700 мс.

**Ожидаемая достоверность**  
Статус заявки меняется только при 100% подтверждении.

#### 6.5.2. Свойство B — Наличие возражений

**Ожидаемое поведение**  
Если хотя бы один участник против — заявка передаётся менеджеру на ручную проверку.

**Ожидаемая производительность**  
Передача на менеджера — до 400 мс.

**Ожидаемая достоверность**  
Заявка не получает статус "выполнено".

### 6.6. Тест 6 — Инициация обмена заявкой

#### 6.6.1. Свойство A — Корректность создания запроса на обмен

**Ожидаемое поведение**  
Участник может отправить предложений обмена, запрос отображается у остальных участников.

**Ожидаемая производительность**  
Создание запроса — до 300 мс.

**Ожидаемая достоверность**  
Запрос корректно отображается всем участникам.

#### 6.6.2. Свойство B — Принятие обмена

**Ожидаемое поведение**  
Другой участник может принять обмен, после чего исполнитель заявки обновляется.

**Ожидаемая производительность**  
Обработка обмена — до 500 мс.

**Ожидаемая достоверность**  
Исполнитель заявки корректно обновлён, история обменов фиксируется.

### 6.7. Тест 7 — Аналитика заявок

#### 6.7.1. Свойство A — Корректность отображения статистики

**Ожидаемое поведение**  
Менеджер видит актуальные данные: количество выполненных заявок, проблемные зоны, рейтинг участников.

**Ожидаемая производительность**  
Время загрузки аналитики — до 800 мс.

**Ожидаемая достоверность**  
Показатели соответствуют фактическим данным.

#### 6.7.2. Свойство B — Пустая аналитика

**Ожидаемое поведение**  
Если заявок и данных нет — система отображает пустые графики и нули, без ошибок.

**Ожидаемая производительность**  
Загрузка — до 300 мс.
